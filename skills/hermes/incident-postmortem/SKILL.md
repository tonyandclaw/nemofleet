# 事故檢討報告模板

## 使用時機
- 服務中斷或重大故障後
- 影響使用者或業務的異常事件
- 需要記錄並分享給團隊/管理層的事故回顧

> 核心理念：Blameless（無責）— 聚焦於系統與流程改善，而非追究個人責任

---

## 報告結構

### 1. 摘要 (Executive Summary)
```markdown
## 摘要

- **事故編號**: INC-YYYYMMDD-XXX
- **事故名稱**: [簡短描述，如：支付服務 API 超時導致交易失敗]
- **發生時間**: YYYY-MM-DD HH:MM (TZ)
- **持續時間**: XX 分鐘 / X 小時
- **影響等級**: P0/P1/P2/P3
- **狀態**: 已解決 / 進行中 / 追蹤改善
- **報告撰寫人**: @username
```

### 2. 影響範圍 (Impact)
```markdown
## 影響範圍

### 影響的服務/系統
- [ ] 服務 A
- [ ] 服務 B

### 影響的使用者
- 估計受影響使用者數：XXX 人 (XX% 活躍使用者)
- 使用者體驗：無法登入 / 交易失敗 / 資料顯示異常 / 效能緩慢

### 業務影響
- 交易損失：$XXX (如有)
- SLA 違約：是 / 否
- 品牌/信任影響：[描述]
```

### 3. 時間線 (Timeline)
```markdown
## 時間線 (所有時間為 TZ)

| 時間 | 事件 | 備註 |
|------|------|------|
| HH:MM | [首次異常徵兆，如：監控告警觸發] | [Alertmanager / PagerDuty] |
| HH:MM | [工程師收到通知並開始調查] | @oncall-engineer |
| HH:MM | [確認問題並開始修復] | |
| HH:MM | [修復完成，服務恢復] | |
| HH:MM | [確認完全恢復，關閉告警] | |

### 關鍵時間點
- **MTTD** (Mean Time To Detect): XX 分鐘
- **MTTR** (Mean Time To Repair): XX 分鐘
```

### 4. 根因分析 (Root Cause Analysis)
```markdown
## 根因分析

### 表層原因 (Symptom)
[描述觀察到的現象，如：資料庫連線池耗盡]

### 深層原因 (Root Cause)
使用 5 Whys 分析：
1. **Why?** [現象] → [直接原因]
2. **Why?** [直接原因] → [上層原因]
3. **Why?** [上層原因] → [系統原因]
4. **Why?** [系統原因] → [流程/設計原因]
5. **Why?** [流程/設計原因] → [根本原因]

### 促成因素 (Contributing Factors)
- [x] [程式碼 Bug / 設定錯誤 / 部署問題 / 容量不足 / 第三方依賴 / 人為操作]
- [x] [監控盲點 / 告警未觸發 / 文件缺失]

### 為何沒有在第一時間發現
[說明監控、測試或流程上的缺口]
```

### 5. 修復過程 (Resolution)
```markdown
## 修復過程

### 採取的行動
1. **第一步**: [描述，如：重啟服務、擴容、回滾版本]
   - 執行人：@username
   - 結果：有效 / 無效 / 部分有效
   
2. **第二步**: [描述]
   - 執行人：@username
   - 結果：有效 / 無效 / 部分有效

### 修復驗證
- [x] 核心功能測試通過
- [x] 監控指標恢復正常
- [x] 使用者回報確認恢復
```

### 6. 預防措施 (Action Items)
```markdown
## 預防措施

### 立即行動 (Immediate - 本週內)
| # | 項目 | 負責人 | 期限 | 狀態 |
|---|------|--------|------|------|
| 1 | [具體可執行事項] | @owner | YYYY-MM-DD | 待辦 |

### 短期改善 (Short-term - 2 週內)
| # | 項目 | 負責人 | 期限 | 狀態 |
|---|------|--------|------|------|
| 1 | [具體可執行事項] | @owner | YYYY-MM-DD | 待辦 |

### 長期改善 (Long-term - 1 個月內)
| # | 項目 | 負責人 | 期限 | 狀態 |
|---|------|--------|------|------|
| 1 | [架構/流程改善] | @owner | YYYY-MM-DD | 待辦 |

### 相關工單
- [JIRA-123] [描述]
- [GitHub Issue #456] [描述]
```

### 7. 經驗教訓 (Lessons Learned)
```markdown
## 經驗教訓

### 做得好的
- [ ] [快速反應、有效溝通、工具使用等]

### 可以更好的
- [ ] [偵測速度、溝通方式、資訊取得等]

### 新發現的風險
- [ ] [描述潛在的類似風險]
```

---

## 使用說明

1. 複製此模板並填入具體內容
2. 所有時間請統一時區，建議使用 UTC 或團隊約定的時區
3. 根因分析使用 5 Whys 確保挖到真正的根本原因
4. Action Items 必須是具體、可執行、有明確負責人和期限的
5. 分享前請相關人員審閱，確認事實正確
6. 報告完成後在團隊會議中分享討論

## 輸出格式建議
- Markdown 文件存於 `/docs/postmortems/INC-YYYYMMDD-description.md`
- 或發布至團隊 Wiki / Notion / Confluence
- 可搭配事故時間線視覺化圖表
