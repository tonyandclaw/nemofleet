# 客戶進度週報 (Customer Progress Report)

## 什麼是客戶進度週報

每週向客戶發布的專案進度報告，目的在於建立透明度、維持信任關係，並確保雙方對專案狀態有一致認知。

## 何時使用

- 需要定期與客戶同步專案進度
- 合約要求每週/雙週提交進度報告
- 專案進入執行階段，有實際產出時

## 不適用情境

- 沒有實質進展時（建議發簡訊通知而非躉報）
- 遇到緊急問題需要即時處理（直接打電話）

## 標準週報流程

**Step 1 — 資料彙整**
- 回顧本週完成的任務（對照方甘特圖或任務清單）
- 收集可展示的產出（截圖、Demo 影片、文件連結）
- 確認需客戶協助的事項

**Step 2 — 草稿撰寫**
- 使用下方模板填入內容
- 聚焦「對客戶有價值的訊息」，而非流水帳
- 風格保持專業但親切

**Step 3 — 內部審核**
- 專案經理或主管確認技術描述是否準確
- 檢查是否有不該揭露的內部資訊
- 確認數據正確性

**Step 4 — 發送給客戶**
- 標題格式：[專案名稱] 週報 - YYYY/MM/DD
- 副本給內部相關人員
- 主動詢問客戶是否有問題

**Step 5 — 追蹤回饋**
- 48 小時內確認客戶是否收到
- 記錄客戶提出的疑問或需求
- 列入下周工作項目

## 週報標準段落結構

**一、執行摘要 (Executive Summary)**
- 1-2 句話說明本週整體狀況
- 使用交通燈號：🟢 順利 / 🟡 有風險 / 🔴 需關注

**二、本週完成項目 (Completed This Week)**
- 用條列式，每項包含：
  - 項目名稱
  - 簡短說明（客戶可理解的語言，避免技術術語）
  - 成果展示（選配：截圖、連結、Demo）

**三、進度總覽 (Progress Overview)**
- 整體進度百分比
- 里程碑狀態對照表
- 預計 vs 實際進度差異說明

**四、本週亮點 (Highlights)**
- 特別值得提的成果
- 客戶可感知的價值
- 團隊付出的額外努力

**五、風險與議題 (Risks & Issues)**
- 可能影響進度的因素
- 需要客戶決定的事項
- 已採取的應對措施
- 再次強調：提出問題時，要同時給建議方案

**六、下週計畫 (Next Week Plan)**
- 明確的交付項目（可驗證的定義）
- 預計完成日期
- 需要客戶提供的資源或回饋

**七、待確認事項 (Pending Items)**
- 等待客戶回覆的問題
- 逾期的確認或素材
- 對專案進度的影響說明

## 撰寫原則

1. **客戶視角**：用客戶熟悉的語言，解釋為什麼這件事對他有價值
2. **誠實透明**：遇到延遲或問題時，主動說明不要隱瞞
3. **視覺化**：善用截圖、進度條、表格讓資訊一目瞭然
4. **可驗證**：成果描述具體到客戶可以檢查確認
5. **起承轉合**：本週做了什麼 → 現在在哪裡 → 下一步要去哪

## 常見錯誤

1. **流水帳式記錄**：「星期一開會、星期二寫程式...」→ 改成「完成會員系統核心功能開發」
2. **過度技術細節**：列出 API 端點或程式架構 → 改成「後端資料同步功能已上線測試環境」
3. **沒有時程資訊**：缺乏預計完成時間 → 永遠加上「預計 X 月 X 日交付」
4. **只報喜不報憂**：隱瞞風險直到爆開 → 提前預警並帶解決方案
5. **發了就算**：沒有確認客戶是否收到或理解 → 主動追蹤並邀請提問

## 驗證清單

- [ ] 標題包含專案名稱與日期
- [ ] 執行摘要有交通燈號狀態
- [ ] 完成項目用客戶語言描述，非技術術語
- [ ] 每個項目有可驗證的交付或連結
- [ ] 風險段落同時包含影響與應對方案
- [ ] 下週計畫有明確日期與交付定義
- [ ] 待確認事項標示對專案的影響程度
