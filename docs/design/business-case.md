# business-case.md — 商業計畫(草稿 v0.1)

> ⚠ 草稿:loop 自動生成的初稿,**數字皆為量級假設,需 Tony / 市場團隊用 ASUS 實際出貨量、財報、人力成本校準**後才可對外。對齊競賽 PPTX slide 12,把那一頁深化成可討論的文件。
> 一句話:把 ASUS 自家網路產品維運的內部能力,**產品化成可隨硬體出貨的「受治理維運勞動力」** —— 不是再一個會建議的 AI,而是敢在治理下動手、修不了會升級人的數位 IT 工。

## 1. 誰會用(市場與客戶)

| 客群 | 對應現成的人 | 為什麼是我們 |
|---|---|---|
| **ASUS 自家網通維運**(首發 dogfood) | 內部 NOC / IT 工程 | 我們有硬體權限、韌體原始碼、設計文件 —— 別人做不到 source SAST + 設計符合性 |
| **商用網通客戶**(ExpertWiFi 商用線、企業 IT) | 服務台 + 網管工程 | 隨硬體出貨、本地可控(資料不離機),企業敢讓它碰維運 |
| **MSP / 託管服務商** | 客服 + 派工工程師 | 一人管更多機隊;前台接客(Hermes)+ 後台機隊(OpenClaw)正好對應其組織 |
| **電信 / 大型邊緣機隊** | 維運中心 | 水平擴展 N 台 OpenClaw,NemoClaw 管生命週期 |

產品結構天然對應企業現成的兩組人:**Hermes = 前台接客**(多通道報修、判型派工、結案),**OpenClaw ×N = 後台 IT 機隊**(監控、實修、升級)。不是要客戶改流程,是把他們現有的人力結構數位化。

## 2. 痛點 → 價值(為什麼買)

維運的痛分兩段(PPTX slide 6):**發現靠運氣**(人工巡檢,一台沉默設備整季沒人發現)+ **處理靠排隊**(報修→轉單→排工程師→修→回報,每步都在等人有空)。

我們把兩段都自動化、且**受治理**:
- 發現:機隊監控(現況 vs 核准基準)+ 定期 CVE 掃描 → 不靠人剛好看到。
- 修復:安全退化當場修回基準(實測 44 秒);一般變更不亂動、列待審。
- 升級:修不了 / 需人工核准 → 自動開 Jira 給工程師(人在迴路),附可合併 patch。
- 治理:每個動作都在 code policy 之下(ALLOWED/DENIED),全程留證據 —— 企業敢讓 AI 動手的前提。

差異:**不是「什麼都敢改」,是只修該修的、變更留審、修不了升級人**。這是「治理下的自動化」,不是放任。

## 3. 市場規模(TAM/SAM/SOM,量級假設待校準)

採三層,由內而外:
- **SOM(起點)**:ASUS 自家網通機隊維運 + 早期商用客戶。以「每台受管設備的維運訂閱」計。
- **SAM**:ASUS 商用網通(ExpertWiFi、商用 AP/路由/伺服器)出貨基數 × 維運自動化滲透率。
- **TAM**:廣義企業/MSP 網路與邊緣設備維運自動化市場(不限 ASUS 硬體,長期可貼牌/開放第三方設備接入)。

> 校準需要:ASUS 商用網通年出貨量、企業/MSP 客戶數、競品(RMM/NMS/ITSM)市場規模。本文不杜撰絕對金額。

護城河:**硬體 + 韌體原始碼 + 設計文件三者整合**。純軟體競品拿不到 source 做 SAST、拿不到設計文件做符合性比對 —— 我們能從「不知道有沒有漏洞」做到「指得出哪一行、附上 patch」。

## 4. 定價模型(三選一或混合)

| 模式 | 計價 | 適合 |
|---|---|---|
| **隨硬體出貨(bundle)** | 嵌入韌體 / 按設備一次性授權 | ASUS 自家、商用網通客戶 —— 把維運力當硬體賣點 |
| **維運訂閱(SaaS-on-prem)** | per-device per-month 或 per-fleet | MSP、企業持續維運 |
| **企業 License** | on-prem 永久授權 + 維護 | 重視資料本地、法遵嚴格的客戶(金融/政府/電信) |

分層(good/better/best):
- **基礎**:監控 + CVE 掃描 + 告警(發現自動化)。
- **進階**:+ 自動修復安全退化 + Jira 升級(處理自動化)。
- **旗艦**:+ 原始碼 SAST + 設計符合性 + 建議 patch(只有拿得到 source 的設備能解鎖)。

## 5. 競品對照(差異化)

| 類別 | 代表 | 它做什麼 | 缺口 = 我們的位置 |
|---|---|---|---|
| 「會建議的 AI」 | ChatGPT / Copilot 類 | 給建議、寫腳本 | **不動手、無治理、不負責驗收** |
| RMM / NMS 網管監控 | 傳統網管平台 | 監控告警 | 不自動修、無 AI 判型派工、無原始碼分析 |
| ITSM 工單系統 | ServiceNow 類 | 開單、流程 | 不動手修,只是把排隊數位化 |
| 雲端 AI agent SaaS | 各家 | 自動化動作 | 資料離機、企業不敢讓它碰生產設備 |
| **本方案** | NemoClaw 四元件 | 監控→判型→**動手修**→驗收→**修不了升級人** | 敢在治理下動手 + 人在迴路 + 隨硬體出貨 + **本地可控** |

一句話差異:別人停在「建議」或「開單」,我們在**政策、授權、驗收全部成立時,才有限度地把該做的事做完,並留下完整證據**。

## 6. ROI 試算(範例;數字為假設,公式可代入客戶實際值)

情境:一個 MSP / 企業管 **200 台** ASUS 網路設備。

| 項目 | 人工 | 雙代理 | 說明 |
|---|---|---|---|
| 例行巡檢 + 設定檢查 | ~100 工時/月(每台 ~30 分/月) | ~10 工時/月(只核准 exception,假設 10% 需人介入) | 機器做、人只核准 |
| 人力成本(假設 NT$800/工時含 overhead) | ~NT$80,000/月 | ~NT$8,000/月 | |
| 平台成本(假設 per-device NT$100/月) | — | ~NT$20,000/月 | 訂閱 |
| **月成本** | **~NT$80,000** | **~NT$28,000** | **節省 ~65%** |
| 安全退化 MTTR | ~7 天(這類退化平均未處理) | **~44 秒**(實測) | 暴露窗口從天級壓到秒級 |

附加價值(難量化但真實):安全退化暴露窗口縮短 → 暴力破解 / 管理介面被攻破 / 查無 log 的風險顯著下降;機隊越大,自動巡檢的邊際效益越高(人工根本掃不完數百台)。

## 7. Go-to-Market(怎麼推)

1. **Phase 1 — 自家 dogfood**:ASUS 內部網通維運先用(已有硬體權限、文件、原始碼優勢);累積真實 MTTR / 攔截 / 開單數據當案例。**已用實機 ASUS ExpertWiFi EBG19P(商用 PoE+ VPN 閘道,韌體 3.0.0.6.102_45537)接入機隊作第三台受管設備驗證** —— 證明新機型上線只需一份核准 baseline + 安全鍵定義,監控/CVE/告警即時涵蓋。
2. **Phase 2 — 隨商用網通出貨**:ExpertWiFi 商用線 / 企業客戶 bundle,把「維運力」當硬體差異化賣點。
3. **Phase 3 — 垂直複製**:MSP、電信、其他 IoT/邊緣設備廠商貼牌;長期開放第三方設備接入(擴大 TAM)。

切入點優先順序遵循「我們最強的地方」:有 source + 設計文件的 ASUS 設備 → 旗艦功能(SAST/符合性/patch)無人能及,先打這塊建立標竿,再外擴。
